In microservices, one service often calls another. Example: Order Service → Payment Service → Bank API If Bank API becomes slow or down, then: Payment service waits Threads get blocked Request queue grows Eventually entire system crashes This is called: 🚨 Cascading Failure Circuit breaker prevents this. Real World Analogy Think of an electrical circuit breaker. When current overload happens: Circuit Breaker → trips → stops electricity This protects devices. Same idea in software. If service keeps failing: Circuit Breaker → stops calling service temporarily Circuit Breaker States There are 3 states. 1. Closed State (Normal) Everything works normally. Service A → Service B → success Requests flow normally. System also monitors: failure count response time 2. Open State (Failure Mode) If failures cross threshold: Example 50% failures in last 10 seconds Circuit breaker opens. Now: Service A → Service B (blocked) Requests fail immediately. Instead return: fallback response cached response error message Example Payment temporarily unavailable This prevents thread exhaustion. 3. Half-Open State (Recovery Check) After some time: sleep window = 30 seconds Circuit breaker allows few test requests. Service A → Service B (test call) If success: Circuit → CLOSED If failure: Circuit → OPEN again Circuit Breaker Flow Request ↓ Circuit Breaker ↓ Service Call Possible outcomes Success → return response Failure threshold reached → open circuit Open circuit → return fallback Simple Flow Diagram Closed ↓ failures Open ↓ timeout Half-Open ↓ success Closed Example Scenario (Interview) Example: Order Service → Inventory Service Inventory service becomes slow. Without circuit breaker: threads waiting ↓ thread pool exhausted ↓ entire order system crashes With circuit breaker: Order Service → Circuit Breaker ↓ fallback response System stays healthy. Circuit Breaker in Java (Resilience4j) Modern Java microservices use Resilience4j. Example: @CircuitBreaker(name = "paymentService", fallbackMethod = "fallback") public String callPaymentService() { return restTemplate.getForObject( "http://payment/api", String.class ); } public String fallback(Exception ex) { return "Payment service temporarily unavailable"; } Configuration example resilience4j: circuitbreaker: instances: paymentService: failureRateThreshold: 50 waitDurationInOpenState: 30s slidingWindowSize: 10 Related Patterns (Often Asked Together) Interviewers often ask these together. Pattern Purpose Retry Retry failed requests Timeout Stop waiting after time Bulkhead Isolate resources Circuit Breaker Prevent cascading failures Fallback Return alternative response Real Production Example Example: Amazon checkout Checkout Service ↓ Recommendation Service If recommendation service fails: Instead of blocking checkout: Circuit Breaker → skip recommendations User can still complete purchase. Senior-Level Interview Answer When interviewer asks: "How do you prevent cascading failures in microservices?" Good answer: "We implement resilience patterns like timeouts, retries, circuit breakers, and bulkhead isolation. Circuit breakers monitor downstream failures and temporarily stop calls when the failure rate crosses a threshold, preventing thread exhaustion and cascading failures." One More Advanced Question They May Ask Follow-up questions might be: Difference between Retry vs Circuit Breaker What is Bulkhead pattern How Hystrix vs Resilience4j works How to tune failure threshold Where to place circuit breaker (client vs gateway) 1. Difference: Retry vs Circuit Breaker Aspect Retry Circuit Breaker Purpose Retry failed request Prevent calling failing service Behavior Re-attempt request Stop requests temporarily Use case Temporary network glitches Service outage Risk Can overload failing service Protects system Example retry 3 times open circuit after failures Example Scenario Retry Flow Order Service → Payment Service Payment fails due to network glitch. System retries. Attempt 1 → Fail Attempt 2 → Success Retry is useful for transient failures. Circuit Breaker Flow Payment service completely down. Without circuit breaker: Order → Payment Order → Payment Order → Payment Threads keep waiting. System crashes. With circuit breaker: Order → Circuit Breaker ↓ fallback response Requests fail fast. Interview Tip Strong answer: Retry handles temporary failures, while circuit breaker protects the system when a dependency becomes consistently unhealthy. In practice we combine: Retry + Circuit Breaker + Timeout 2. Bulkhead Pattern Bulkhead pattern isolates system resources so one failure does not affect the whole system. Real World Analogy Ships have separate compartments (bulkheads). If one compartment floods: Only that section floods Ship does not sink Software Example Example service: Ecommerce Service Calls: Payment Service Recommendation Service Notification Service Without bulkhead: Single thread pool If recommendation service becomes slow: All threads blocked Checkout also stops With Bulkhead Separate resources. ThreadPool1 → Payment ThreadPool2 → Recommendation ThreadPool3 → Notification If recommendation fails: Checkout still works Java Example (Resilience4j) @Bulkhead(name = "paymentService", type = Bulkhead.Type.THREADPOOL) public String processPayment() { return paymentService.call(); } 3. Hystrix vs Resilience4j Hystrix used to be the Netflix standard, but now Resilience4j replaced it. Feature Hystrix Resilience4j Maintained ❌ No ✅ Yes Library size Heavy Lightweight Programming model Thread isolation Functional Spring Boot support Old Modern Performance Lower Higher Hystrix Example @HystrixCommand(fallbackMethod = "fallback") public String callPayment() { return restTemplate.getForObject(url, String.class); } Resilience4j Example @CircuitBreaker(name="paymentService", fallbackMethod="fallback") public String callPayment() { return restTemplate.getForObject(url, String.class); } Why Industry Moved to Resilience4j Reasons: modular architecture better performance supports reactive programming integrates with Spring Boot 2+ 4. How to Tune Failure Threshold Circuit breaker triggers based on failure rate. Common configuration: failureRateThreshold: 50 slidingWindowSize: 20 waitDurationInOpenState: 30s Meaning Example: Last 20 requests monitored If 10 failures → 50% Circuit opens. Key Parameters Failure Rate Threshold Typical value 50% – 60% Sliding Window Size Number of requests monitored. Example 20 requests 50 requests 100 requests Open State Duration How long circuit remains open. Example 30 seconds After that Half-open state How Companies Tune It Based on: SLA error rate service latency traffic patterns Example: Payment service → strict threshold Recommendation service → relaxed threshold 5. Where to Place Circuit Breaker Two options. Option 1 — Client Side (Most Common) Service calling another service applies circuit breaker. Example Order Service → Circuit Breaker → Payment Service Advantages fine control service specific configuration Used in Spring Boot + Resilience4j Option 2 — API Gateway Level Gateway protects backend services. Example Client ↓ API Gateway (Circuit Breaker) ↓ Microservices Used in Spring Cloud Gateway Kong Envoy Best Practice Use both. Gateway → coarse protection Service → fine protection 6. Circuit Breaker (Resilience4j) Dependency io.github.resilience4j resilience4j-spring-boot3 Service Code @Service public class PaymentServiceClient { @CircuitBreaker(name = "paymentService", fallbackMethod = "fallback") public String callPaymentService() { return restTemplate.getForObject( "http://payment-service/api/pay", String.class ); } public String fallback(Exception ex) { return "Payment service temporarily unavailable"; } } Configuration resilience4j: circuitbreaker: instances: paymentService: failureRateThreshold: 50 slidingWindowSize: 10 waitDurationInOpenState: 30s 2. Retry Pattern Retries failed requests automatically. Code @Service public class PaymentRetryService { @Retry(name = "paymentRetry", fallbackMethod = "fallback") public String processPayment() { return restTemplate.getForObject( "http://payment-service/api/pay", String.class ); } public String fallback(Exception ex) { return "Retry failed"; } } Config resilience4j: retry: instances: paymentRetry: maxAttempts: 3 waitDuration: 2s Flow Request ↓ Fail ↓ Retry ↓ Retry again ↓ Fallback 3. Timeout (TimeLimiter) Prevents requests from waiting forever. Code @Service public class PaymentTimeoutService { @TimeLimiter(name = "paymentTimeout") public CompletableFuture callPayment() { return CompletableFuture.supplyAsync(() -> restTemplate.getForObject( "http://payment-service/api/pay", String.class ) ); } } Config resilience4j: timelimiter: instances: paymentTimeout: timeoutDuration: 2s If service takes >2 seconds → request fails. 4. Bulkhead Pattern Limits number of concurrent calls. Code @Service public class PaymentBulkheadService { @Bulkhead(name = "paymentBulkhead", type = Bulkhead.Type.THREADPOOL) public String processPayment() { return restTemplate.getForObject( "http://payment-service/api/pay", String.class ); } } Config resilience4j: bulkhead: instances: paymentBulkhead: maxConcurrentCalls: 10 Meaning Only 10 requests allowed simultaneously Extra requests rejected. 5. Rate Limiter (Resilience4j) Limits number of requests. Code @Service public class PaymentRateLimiterService { @RateLimiter(name = "paymentRateLimiter") public String callPayment() { return restTemplate.getForObject( "http://payment-service/api/pay", String.class ); } } Config resilience4j: ratelimiter: instances: paymentRateLimiter: limitForPeriod: 10 limitRefreshPeriod: 1s Meaning 10 requests allowed per second 6.
