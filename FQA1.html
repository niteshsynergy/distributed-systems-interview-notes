Communication Types – Synchronous vs Asynchronous 1.1 Synchronous Communication Definition Client sends request → waits for response. Example REST API call Database query Blocking HTTP request Spring Example Using RestTemplate RestTemplate restTemplate = new RestTemplate(); String response = restTemplate.getForObject( "http://orderservice/order/1", String.class ); Characteristics Feature Value Blocking Yes Latency Higher Complexity Low Error Handling Easy When to use Simple services Small latency Immediate response required 1.2 Asynchronous Communication Definition Client sends request → continues execution → result comes later. Examples Kafka RabbitMQ Async HTTP calls CompletableFuture Spring WebClient Example WebClient client = WebClient.create(); Mono response = client.get() .uri("http://orderservice/order/1") .retrieve() .bodyToMono(String.class); response.subscribe(System.out::println); Characteristics Feature Value Blocking No Latency Low Throughput High Complexity High Used in Distributed systems High traffic systems Microservices communication 2. Parallel Calls Interviewers often ask: How do you call multiple services in parallel? 3.1 Callable + Future Example ExecutorService executor = Executors.newFixedThreadPool(5); Callable task = () -> { return restTemplate.getForObject("http://service", String.class); }; Future future = executor.submit(task); String result = future.get(); Problem Blocking Hard to manage multiple calls 3.2 CompletableFuture (Best Practice) Example CompletableFuture service1 = CompletableFuture.supplyAsync(() -> callService1()); CompletableFuture service2 = CompletableFuture.supplyAsync(() -> callService2()); CompletableFuture service3 = CompletableFuture.supplyAsync(() -> callService3()); CompletableFuture.allOf(service1, service2, service3).join(); String result = service1.get() + service2.get() + service3.get(); Advantages Parallel execution Non-blocking Easy aggregation 4. Multiple CallableFuture Use Case Example scenario User Dashboard API Needs Order service Payment service Recommendation service Sequential API → Service1 → Service2 → Service3 Latency 100 + 100 + 100 = 300ms Parallel API → (S1,S2,S3) parallel Latency max(100,100,100) = 100ms Performance improvement ~66% faster 5. How Payment Systems Prevent Double Debit This is one of the most important fintech interview questions. Example problem: User clicks PAY twice Network retry happens API retry happens Without protection Account debited twice Method 1 — Idempotency Key (Most Common) Client sends a unique id for each payment. Example POST /payment Headers: Idempotency-Key: txn-789123 Server logic if txnId already processed return previous result else process transaction Database txn_id UNIQUE So duplicate requests return same response. Method 2 — Database Unique Constraint Example table payments --------- txn_id (unique) user_id amount status If duplicate request comes DB rejects insert Method 3 — Distributed Lock Used when multiple servers process same transaction. Example using Redis. SETNX payment_123 lock Only one server gets lock. Others fail. Method 4 — Two Phase Commit / Transaction Banks use ACID transactions. Example BEGIN TRANSACTION Debit account A Credit account B COMMIT If failure ROLLBACK Method 5 — Payment State Machine Payments follow strict states. INITIATED PROCESSING SUCCESS FAILED If status already SUCCESS system ignores duplicate request. Real Industry Example UPI / Stripe / PayPal all use Idempotency Key + DB unique constraint This guarantees exactly once payment processing. 3. System Design for Rate Limiter Very common system design question. Goal Limit requests per user. Example 100 requests per minute per user High Level Architecture img img_1,2,3 read Client ↓ API Gateway ↓ Rate Limiter (Redis) ↓ Application Server Common Rate Limiting Algorithms 1. Fixed Window Example 100 requests / minute Implementation key = userId:timestamp count++ Stored in Redis. Problem Burst allowed at boundary. Example 100 requests at 59s 100 requests at 1s Total 200. 2. Sliding Window Improvement. Track last N seconds. Example last 60 seconds requests Better accuracy. But memory expensive. 3. Token Bucket (Most Popular) Tokens refill at fixed rate. Example capacity = 100 refill = 10 tokens/sec Flow Request arrives ↓ If token available → allow Else → reject Used by AWS Stripe Google APIs Example Redis Implementation Key rate_limit:user123 Value remaining_tokens timestamp Pseudo code if tokens > 0 tokens-- allow request else reject request Where Rate Limiter Is Implemented Usually at API Gateway Examples Kong NGINX AWS API Gateway Envoy Handling 1 Crore Requests with Rate Limiter Use distributed rate limiter. Architecture Clients ↓ API Gateway cluster ↓ Redis Cluster ↓ Services Redis ensures global rate limits across all servers. Enforce limits at API Gateway 5. Use fallback if Redis unavailable One Powerful Interview Tip (10+ Years) When answering system design questions, always mention: Scalability Fault tolerance Consistency Observability Example: "We would use Redis cluster for distributed rate limiting, ensure horizontal scalability via API gateway instances, and add metrics via Prometheus to monitor rejection rates."
